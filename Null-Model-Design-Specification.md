# 空模型范式（NMP）设计规范

> **版本**: 2.0.0  
> **日期**: 2026-07-18  
> **定位**: 本地私域 RAG 引擎——TDA 架构第一层（先验计算结构）的参考实现  
> **状态**: 已实现，可独立部署  
> **配套代码**: NMP 参考实现 v4.0.0（nmp_ui.py / nmp.py / crash_probe.py）

## 前言
>
> ***—— 知识是动力，目的是方向，而事实是方向盘***。
>
>***—— 事实是判断的依据***。

#

OpenAI最新发表并被称为"开创性论文"的《*Why Language Models Hallucinate*》https://arxiv.org/abs/2509.04664

洋洋洒洒貌似精辟地分析了各种原因，并号称给出了解决方向，但这些分析，本质上仍然停留在"知其然"的层面，而没有真正回答"为什么会这样"的根本问题，也因此根本没有给出真正解决问题的具体办法——换言之，给出的办法根本没有解决问题。

其实，这个根本原因，一句话就能说清楚：  
> **大模型是抛开事实讲逻辑、讲道理。**

抛开事实讲逻辑、讲道理，能不胡说八道吗？能不产生幻觉吗？能不进行逻辑脑补吗？***判断，以事实为依据，这不是常识吗***？

原因如此简单。

解决办法一样很简单：**那就给大模型事实**。

## 一、核心定义

### 1.1 空模型是什么

**空模型 = 本地私域 RAG 引擎**

- **输入**：用户语义（自然语言问题）
- **处理**：向量检索（语义编码 → 混合检索 → 重排序）
- **输出**：{用户问题, 相关事实列表}（结构化语义包）
- **本质**：不做意图解析、不构建约束、不执行校验、不调用 AI 模型

> **"空"的含义**：空掉内容（无领域知识、无训练数据、无价值偏好），留下结构（向量检索骨架）。像 CPU，有指令集，无数据；像镜子，能映照，无偏见。

### 1.2 空模型不是什么

| 不是 | 说明 |
|:---|:---|
| **不是 AI 模型** | 无神经网络推理、无生成能力、无概率输出 |
| **不是规则引擎** | 不执行硬编码规则，只做语义匹配 |
| **不是知识图谱** | 不存储世界知识，只检索使用者确认的事实 |
| **不是约束构建器** | 不生成 "no_touch"、"max_speed" 等约束，只输出原始事实 |
| **不是校验器** | 不判断 LLM 输出对错，只提供事实供 LLM 自主判断 |
| **不是通用 RAG** | 不检索外部网页/文档，只检索本地元事实库 |

### 1.3 与通用 RAG 的根本区别

| 维度 | 通用 RAG | NMP 空模型 |
|:---|:---|:---|
| **检索对象** | 外部知识库、网页、文档 | **本地元事实库** |
| **数据性质** | 概率性知识（可能过时、错误） | **确定性事实（使用者确认、有时间戳）** |
| **结果用途** | 参考材料（LLM 可忽略） | **事实约束（LLM 不可逾越底线）** |
| **更新机制** | 自动爬取/定期同步 | **人工 OTA 确认** |
| **所有权** | 公共/企业 | **使用者私域** |
| **延迟要求** | 可接受秒级 | **<10ms（本地优先）** |

## 二、核心流程

```
用户问题（自然语言）
    ↓
┌───────────────────────────────┐
│  语义编码层                    │
│  用户问题 → 查询向量            │
│  模型:sentence-transformers    │
└───────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│  混合检索层                          │
│  ├── 稠密检索：FAISS 向量相似度       │
│  ├── 稀疏检索：BM25 / SQLite FTS     │
│  └── 图检索：因果骨架邻接遍历（预留)   │
└─────────────────────────────────────┘
    ↓
┌────────────────────────────┐
│  重排序层                   │
│  ├── 语义相关性打分         │
│  ├── 时效性衰减（时间戳权重）│
│  └── 使用者偏好（历史反馈）  │
└────────────────────────────┘
    ↓
┌──────────────────────────────────────────────────────────────────┐
│  事实打包层                                                       │
│  输出结构化语义包：                                                │
│  {                                                               │
│    "question": "用户问题",                                        │
│    "facts": [                                                    │
│      {"fact_id": "F001", "content": "...", "score": 0.95},       │
│      {"fact_id": "F002", "content": "...", "score": 0.88}        │
│    ],                                                            │
│    "uncertainty": ["未获取到..."]  // 可选                        │
│  }                                                               │
└──────────────────────────────────────────────────────────────────┘
    ↓
LLM 接收语义包，自主约束/自由生成
```

## 三、与周边组件的关系

### 3.1 与元事实库

| 元事实库 | 空模型 |
|:---|:---|
| **角色** | 数据存储 | 计算引擎 |
| **内容** | 使用者确认的静态事实快照 | 无内容，只读取 |
| **职责** | 持久化、版本管理、权限控制 | 检索、筛选、打包 |
| **接口** | SQL 查询、向量检索 | 调用元事实库接口 |
| **可见性** | 对 LLM 完全不可见 | 对 LLM 完全不可见 |

**关键原则**：空模型只读元事实库，不修改、不生成新事实。

### 3.2 与 LLM

| LLM | 空模型 |
|:---|:---|
| **角色** | 策略生成器 | 事实提取器 |
| **能力** | 推理、创作、规划 | 检索、匹配、打包 |
| **权限** | 在事实约束下生成 | 无权修改事实，无权直接执行 |
| **关系** | 接收事实包，自主判断 | 为 LLM 提供判断依据 |

**关键原则**：空模型不约束 LLM，只提供事实。约束来自：
- **Prompt 设计**（人写的指令）
- **LLM 自身对齐**（训练时的价值观）
- **后置校验**（可选，如 nmp.py 的冲突检测）

### 3.3 与判例库

| 判例库 | 空模型 |
|:---|:---|
| **角色** | 否定性知识档案 | 检索引擎 |
| **内容** | 历史错误记录 | 不存储，只匹配（当前版本） |
| **关系** | 接收空模型输出的反馈历史 | 当前版本不直接查询判例库 |

> **说明**：当前 NMP 版本（v4.0.0）中，空模型不直接查询判例库。判例库用于外部闭环（反馈 → 判例积累 → 裁决 AI 调阈值 → 影响空模型检索参数）。TDA 完整版中，判例库将集成进检索流程。

### 3.4 与探针

| 探针 | 空模型 |
|:---|:---|
| **角色** | 外部事实提取工具 | RAG 引擎 |
| **关系** | 探针输出 → 存入元事实库 → 空模型检索 | 不直接调用探针 |
| **开发者** | 使用者/行业开发者自行开发 | 通用框架 |

**示例探针**：
- `crash_probe.py`：硬件探针（CPU/内存/GPU/进程/日志/转储）
- 软件探针：应用日志解析、配置读取
- 网络探针：延迟测试、带宽检测
- 传感器探针：温度/湿度/压力读取
- 业务探针：金融交易记录、医疗病历导入

## 四、元事实库规范

### 4.1 核心原则

> **没有公共通用的事实，只有个体私域的事实。**

- **没有无目的的事实**：事实只在特定使用者的特定目的下被提取
- **没有无主人的事实**：事实属于确认它的使用者，LLM 只代表使用者说话
- **没有无时间戳的事实**：每个事实都是某个时间刻度上的静态快照

### 4.2 数据结构

```json
{
  "vault_id": "user_master_v1",
  "version": "2026.06.26",
  "owner": "使用者身份标识",
  "owner_type": "individual|enterprise|organization",
  "purposes": ["health_reminder", "anomaly_detection"],
  "facts": [
    {
      "fact_id": "F001",
      "content": {
        "entity": "medical_history",
        "attributes": {
          "condition": "gastritis",
          "severity": "moderate",
          "diagnosed_date": "2024-03-15"
        }
      },
      "type": "personal_fact",
      "confirmed_by": "user",
      "confirmed_at": "2024-03-15T10:00:00",
      "source": "hospital_record_import",
      "status": "active",
      "effective_from": "2024-03-15",
      "effective_until": null,
      "superseded_by": null,
      "version": 1
    }
  ],
  "constraints": [],
  "ontology_mappings": [],
  "meta_facts": [],
  "logs": []
}
```

**关键字段**：
- `owner`：事实主权者
- `confirmed_by`：确认者身份
- `confirmed_at`：确认时间戳
- `status`：`active` / `historical` / `revoked`
- `version`：版本号（变化不是修改，是新增快照）

### 4.3 与 LLM 的隔离

**元事实库对 LLM 完全不可见。**

LLM：
- 不知道存在哪些事实
- 不知道哪些事实被过滤
- 不知道元事实库的存在
- 只接收**脱敏后的结构化事实**

这是架构防火墙：即使 LLM 被攻击或越狱，也无法绕过元事实库的约束——因为约束在**本地空模型**中执行，不在云端。

## 五、判例库规范

### 5.1 核心定义

> **判例库 = 历史错误档案 = 否定性知识**

- 不是"正确答案集"，是"错误病理档案"
- 每一例被拦截的错误，都在增强系统对"错"的识别精度
- 判例库是**私域数据**，非通用、非公共

### 5.2 数据结构

```json
{
  "case_id": "CASE-2024-015",
  "pattern": {
    "object": "电水壶",
    "state": "高温",
    "action": "直接接触"
  },
  "action": "INTERCEPT",
  "priority": "CRITICAL",
  "description": "高温液体直接接触导致烫伤",
  "source": "user_feedback",
  "created_at": "2024-05-20T10:00:00",
  "frequency": 3,
  "related_facts": ["F002"]
}
```

### 5.3 判例库的内部机制（预留）

当前 NMP 版本（v4.0.0）中，判例库为**静态匹配**（模式匹配 + 优先级排序）。

TDA 进化版中，判例库将引入：
- **贝叶斯状态机**：从历史错误中辨识错误类型（状态分类）
- **动态阈值**：基于历史先验分布调节拦截阈值

> **说明**：判例库的贝叶斯-卡尔曼-PID 机制属于 TDA 进化版，当前 NMP 版本仅保留静态判例匹配。

## 七、与 TDA 的关系

### 7.1 两个阶段

| 阶段 | 名称 | 状态 | 能力 |
|:---|:---|:---|:---|
| **初级阶段** | **NMP 空模型** | ✅ 已实现 | 本地私域 RAG 引擎：语义输入 → 语义输出 |
| **进化完整版** | **TDA 三层架构** | ⚠️ 理论设计 | 能动极 + 受动极模型 + 元受动极裁决 |

### 7.2 演进路径

```
NMP 空模型（v2.0，当前）
    ↓ 能力扩展
TDA 第一层（先验计算结构）
    ↓ 增加辩论能力
TDA 第二层（结构化辩论层）—— 受动极模型（PPM）
    ↓ 增加裁决能力
TDA 第三层（元受动极层）—— 裁决 AI + 判例库（贝叶斯-卡尔曼-PID）
```

### 7.3 关键区别

| 维度 | NMP 空模型 | TDA 受动极模型（PPM） |
|:---|:---|:---|
| **能力** | 向量检索 + 事实打包 | 语义理解 + 论证重构 + 冲突图生成 |
| **与 LLM 关系** | 单向输出（事实包） | 双向对抗（并行启动，辩论质证） |
| **智能层级** | 无 AI 模型 | 独立训练的判别/约束型模型 |
| **延迟** | <10ms | 未知（可能 >100ms） |
| **状态** | 已实现 | 待社区共建 |

> **TDA 将单独成篇**，详细阐述三层架构、非对称双视角训练、结构化辩论、元受动极裁决等理论设计。

## 九、核心主张

> **AI 永远不完美，但我们可以用不完美的组件，组合出一个负责任的系统。**

空模型不试图让 LLM "更聪明"，而是做一件更朴素的事：

**给 LLM 一个锚——不是锚定在某个正确答案上，而是锚定在使用者确认的事实上。**

LLM 可以发散、可以创意、可以调用全量知识，但它必须知道：
- **这是谁的事实**（owner）
- **什么时候的事实**（timestamp）
- **什么场景下的事实**（purpose）

空模型只负责把这三个"什么"送到 LLM 面前，剩下的判断，交给 LLM 自己。
